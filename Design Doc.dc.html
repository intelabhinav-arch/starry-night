Starry Fiddler · a calm space fidget for young astronomers

Product & Engineering Design Document

Starry Fiddler v0.1 · Draft

A calm, non-competitive space “fiddler” for children ages 3–5. A soothing dark sky where celestial objects drift by, and every key press produces a gentle, delightful reaction — while grown-ups keep the computer safe.

Author: Design team Date: July 4, 2026 Status: Prototype complete → planning production build Platform: Web (desktop-first)

1 · Vision & Goals

Starry Fiddler is the screen a parent switches on when a young child wants to “use the computer.” It turns keyboard mashing into a serene, wonder-filled space scene rather than a frantic game. The experience rewards cause and effect — the core early-learning loop for this age — while gently seeding letter, number, and planet recognition.

Design principles

  • Calm over rush. Slow drift, soft motion, no timers, no score, no fail state. Nothing chases the child.
  • Every press is a “yes.” Any key does something pleasant. Exact matches are a bonus, never a requirement.
  • Soothing on the eyes. Deep, low-saturation dark sky; adjustable brightness that caps at a comfortable maximum.
  • Grown-ups in control, hidden from kids. Settings live behind a typed passphrase so little fingers can’t reach them.
  • Learning by delight. Planets say their names; letters and numbers appear big and friendly. No lessons, no drills.

Non-goals

  • Not a typing tutor or a structured curriculum.
  • Not a scored or timed game; no win/lose, no levels to beat.
  • Not a full OS lockdown — the hosting app is responsible for kiosk/fullscreen safety (see §8).

2 · Audience & Context

Primary user

Child, 3–5 years. Pre-reader or early reader. Motor control still developing — large targets, forgiving input, no precision required.

Secondary user

Parent/caregiver. Wants a safe, quiet, ad-free activity they can start in seconds and supervise casually. Configures once via the grown-up panel.

Research basis: at this age children think in images and symbols and learn primarily through immediate cause-and-effect and repetition. The design deliberately avoids abstract goals and instead maximizes instant, varied, positive feedback.

3 · Core Experience

A fullscreen dark sky holds distant twinkling stars and faint galaxies. Celestial objects enter from one side and drift slowly across. Many carry a large letter or number badge. The child can:

  • Press the matching key → that object performs its signature reaction (a “celebration” of its own).
  • Press any other key → a big, soft, colored letter floats up with a gentle chime.
  • Click / tap an object → same as pressing its key (touch and mouse friendly).

Every ~30 presses, a gentle supernova celebration blooms in the center with the child’s name (if set). That is the only “milestone,” and it is celebratory, not evaluative.

5 · Visual & Audio Design

Style system

Three interchangeable object treatments, assignable per-object at random or forced globally from the grown-up panel:

  • Soft cartoon — flat pastel fills, rounded shapes, no glow.
  • Glowy neon — outlined shapes with soft light on near-black.
  • Soft + glow — pastel fills with a gentle halo (default recommendation).

Fredoka (rounded, friendly) for object badges, big letters, and headings; Nunito for UI copy in the grown-up panel.

Sound

All audio is synthesized at runtime (Web Audio API) — no asset files. Sine-based chimes, filtered-noise whooshes/pops, and a rising “zuup.” Planet names use the browser’s speech synthesis. A single toggle mutes everything. Audio context resumes on first key press to satisfy autoplay policies.

6 · Grown-up Panel

Opened by typing the word PARENT on the keyboard (invisible to a mashing child). Dismissed with Esc or the close button. All settings persist to localStorage.

7 · Technical Architecture

The prototype is a single self-contained component: a fixed-position scene, a requestAnimationFrame loop advancing all objects and particles, and a keyboard/pointer handler. Recommended structure for the production build:

Scene loop

Single rAF driver. Delta-time integration so motion is frame-rate independent. Spawns a new object on an interval, capped by the busyness setting.

Object model

Plain data records (type, x, y, vx, vy, label, style, mode…) in one array. A sprite(type) renderer maps each to visuals; a hit(type) handler maps each to its reaction.

Particles

Separate lightweight array for sparks, new stars, and rings. Aged out by lifetime. Kept small for performance.

Input

Global keydown: passphrase buffer, exact-match lookup, fallback floating letter. Pointer events on objects mirror key hits.

Audio

One shared AudioContext; small synth helpers per sound. Resumed lazily. Guarded by the sound toggle.

Settings

Single settings object persisted to localStorage; read on load, written on every change.

Suggested refactor for code work: extract object definitions into a declarative registry — each type as { label, spawn, sprite, onHit, sound } — so new floaters (space station, black hole, aurora, planet moons) can be added as data without touching the loop. This is the natural next milestone.

8 · Safety & Deployment

  • Keyboard containment is the hosting app’s job. A plain web page cannot block OS shortcuts (⌘-Space, Alt-Tab, Windows key). Wrap in a kiosk shell (Electron/Tauri, or iPad Guided Access / Android screen pinning) for a true lockdown.
  • Fullscreen reduces accidental exits; offer it prominently on launch.
  • Privacy: no accounts, no network calls, no analytics required. All state is local. Keep it that way for a children’s product.
  • Distribution: ship the component plus its runtime together; the page is offline-capable and can be bundled to a single file.
